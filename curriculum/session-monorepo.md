# 特別回：モノレポ構成とコード共有

## 1. 今日の目標
- モノレポ（Monorepo）の概念とメリットを理解する
- `npm workspaces` を使って、フロントエンドとバックエンドを一つのプロジェクトとして管理する
- Zodのスキーマを「共有パッケージ」に切り出し、型安全性をさらに高める

## 2. モノレポとは？
一つのリポジトリの中に、複数のパッケージ（バックエンド、フロントエンド、共通ライブラリなど）を共存させる構成です。

### メリット
- **型共有が容易**: バックエンドで定義したZodスキーマを、フロントエンドでも直接インポートして使える。
- **一括管理**: 依存関係のアップデートやビルドを一括で行える。
- **変更の一貫性**: APIの仕様変更とフロントエンドの修正を、一つのコミットで完結できる。

## 3. ディレクトリ構成の例
```text
my-project/
├── package.json (ルート)
└── packages/
    ├── backend (Hono)
    ├── frontend (Vite)
    └── shared (Zod スキーマなどの共通定義)
```

## 4. npm workspaces の設定
ルートの `package.json` で、どこにパッケージがあるかを指定します。

```json
{
  "name": "my-monorepo",
  "private": true,
  "workspaces": [
    "packages/*"
  ]
}
```

## 5. ハンズオン：共通パッケージの作成
1. `packages/shared` ディレクトリを作成し、`package.json` を用意します。
2. Zodのスキーマを `packages/shared/src/index.ts` に書きます。
3. `package.json` の `name` を `@my-app/shared` とします。
4. フロントエンドとバックエンドの `package.json` で `"@my-app/shared": "*"` を追加して `npm install` します。

## 6. まとめ
- モノレポは「境界を越えた型安全」を実現するための強力な武器。
- `npm workspaces` を使うことで、複雑な設定なしに複数のパッケージを扱える。
- 特に Zod を共有することで、フロントとバックの「入力チェックの不一致」を完全に防ぐことができる。
